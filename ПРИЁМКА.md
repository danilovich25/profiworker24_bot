# Приёмка бота ProfiWorker24

Документ фиксирует выполнение двух этапов и результаты проверки на рабочем
Telegram-боте `@Profiworker24_manager_bot` и портале Bitrix24 `b24-5o4ele`.

## Этап 1

| Пункт ТЗ | Что сделано | Как проверено |
|---|---|---|
| Бот на aiogram, главное меню и приём текстовых сообщений | Бот работает на aiogram 3 в режиме long polling. В меню есть «Новая заявка», «Найти» и «Последние», зарегистрированы команды `/new`, `/find`, `/last`, `/help`. | Через рабочий Telegram-бот отправлена свободная текстовая заявка. После подтверждения создана сделка №42. Кнопки меню проверены в том же диалоге. |
| Разбор свободной фразы и уточняющие вопросы | Сообщение разбирается в имя, телефон, организацию, адрес, категорию, описание, срок и суммы. Если нет телефона или категории, бот спрашивает их отдельно; при недоступном разборе включает опрос из шести шагов. | Сделка №42 создана из одной свободной фразы. Отдельно проверена смена контекста во время вопроса о телефоне: новая фраза с номером была разобрана заново и создала сделку Ольги №44, не смешавшись со старым черновиком. |
| Интеграция с Bitrix24 через входящий вебхук | Вся работа с CRM выполняется через REST: контакты, сделки, поиск дублей, стадии, пользовательские поля и задачи. При старте бот проверяет обязательные поля и отключает запись в CRM, если портал не готов. | Через рабочий вебхук созданы сделки №42, №44, №46 и №48. Все четыре появились в боевом Bitrix24. |
| Поиск клиента по телефону и организации, создание нового и определение повторного | Перед созданием контакта бот ищет совпадение по телефону. Организация хранится в поле контакта `UF_CRM_ORG`. Повторное обращение использует существующий контакт; защита от повторной доставки и повторного нажатия не даёт создать случайный дубль. | `/find` проверен по телефону, организации и номеру заявки. Запрос `/find Ромашка` находит сделку №46; у её контакта Сергей Иванов заполнено `UF_CRM_ORG=ООО Ромашка`. Повторные сообщения и кнопки проверены без появления второй сделки. |
| Создание сделки с привязкой к контакту и базовыми полями | Сделка получает контакт, понятное название из категории и описания, комментарий с адресом, сроком и организацией, а также начальную стадию `NEW`. Номер в ответе Telegram совпадает с ID сделки. | В сделке №42 указан контакт Иван Петров, телефон `+79141234567`, категория «сантехника» и описание работы. Сделка находится на стадии «Новая заявка». |
| Воронка «Услуги на дому» и семь статусов | Используется стандартная воронка категории `0`: «Новая заявка», «Связались с клиентом», «Назначен исполнитель», «В работе», «Ожидание оплаты», «Завершено», «Отмена». Бот помещает новые сделки в первую стадию `NEW`. | Состав и порядок стадий сверены в рабочем портале. Сделка №42 автоматически попала в «Новую заявку». |
| Доход, расход, прибыль и категория услуги | Доход записывается в штатную сумму сделки `OPPORTUNITY`; расход — в `UF_CRM_EXPENSE`; прибыль — в `UF_CRM_PROFIT`; категория — в `UF_CRM_SERVICE_CATEGORY`. Обязательные поля создаются ботом при старте. | В сделке №42 записаны доход 5000 руб., расход 2000 руб. и прибыль 3000 руб.; поле категории услуги содержит «сантехника». |
| Развёртывание на Railway, README и переменные окружения | Проект собирается из `Dockerfile`, работает на Railway владельца, хранит SQLite на томе `/data`. В README перечислены функции, API и все переменные; отдельные инструкции описывают Railway и Bitrix24. | Рабочий бот доступен в Telegram после деплоя на Railway. Настройки сопоставлены с `app/config.py`, `Dockerfile` и `railway.toml`. |
| Источник заявки (раунд 2) | Отдельное поле «Источник»: Авито, Форпост, Сарафанное радио, Прочее. Пишется в родное поле сделки `SOURCE_ID`; справочник источников портала бот дополняет сам при старте. Источник берётся из текста («по авито», «по сарафану»), в пошаговом опроснике — вопрос с 4 кнопками; если не назван — «Прочее». | Живой прогон 21.07: сделка №60 из фразы с «по авито» получила `SOURCE_ID=AVITO`; голосовая заявка №70 «клиент с авито» — тоже. Справочник портала сверен по `crm.status.list`. Тестовые сделки удалены после проверки. |
| Разбор фразы без запятых (раунд 2) | Сплошная строка без знаков препинания раскладывается по полям так же, как обычная. | Живой прогон 21.07: «Иван 89141234567 сантехника замена крана завтра доход 5000» разложена полностью (имя, телефон, категория, описание, срок, доход). |
| Относительные даты и формат дд.мм.гггг чч:мм (раунд 2) | «Завтра», «послезавтра», «через 3 дня», «через неделю», «через 5 дней в 10:00», дни недели и «24.07.2026 10:00» пересчитываются кодом от текущего времени Владивостока (`TZ`, одна настройка); все даты показываются как `дд.мм.гггг чч:мм`, время из зон других порталов конвертируется. | Живой прогон 21.07: «через 5 дней в 10:00» → 26.07.2026 10:00; «послезавтра в 10 утра» (голосом) → 23.07.2026 10:00; дата создания сделки в списках показана во владивостокском времени. |
| Напоминания в два канала (раунд 2) | У сделки со сроком: ДЕЛО Bitrix24 с напоминанием в момент срока (push мобильного приложения, ответственный — пользователь портала) плюс Telegram-сообщение бота ровно в срок. Очередь напоминаний хранится в SQLite и переживает перезапуск. | Живой прогон 21.07: напоминания по сделкам №62 и №60 пришли в Telegram в 09:08:01 и 09:10:01 при сроках 09:08 и 09:10 — секунда в секунду, включая срок, перенесённый правкой. Дела CRM сверены по REST: дедлайн верный, ответственный id=1. |
| Отмена ввода (раунд 2) | Кнопка «❌ Отмена» на каждом шаге уточнений и опросника, команда `/cancel`. После отмены тот же текст можно прислать заново — дублем не считается. | Живой прогон 21.07: отмена на вопросе о телефоне, повтор той же фразы принят без предупреждения о дубле; `/cancel` вне диалога отвечает «Сейчас нечего отменять». |
| Поиск без лишнего шага (раунд 2) | После «Найти» поле ввода открывается сразу (ForceReply с подсказкой), отдельная кнопка не нужна. | Живой прогон 21.07: ответ на «Найти» пришёл с открытым полем ввода и плейсхолдером «Телефон, № заявки, имя или организация». |
| Редактирование заявки (раунд 2) | Из результатов поиска заявка открывается кнопкой «Открыть №N»: карточка со всеми полями, «✏️ Изменить» — правка полей кнопками, «💾 Сохранить» обновляет ТУ ЖЕ сделку (и контакт) по тем же ID; смена срока переносит дело CRM и Telegram-напоминание. | Живой прогон 21.07: у сделки №60 срок изменён правкой — ID не изменился, дубль не появился, дело перенесено; у №68 после переноса заголовок дела цел («Заявка №68: …»), дедлайн и ответственный верны. |

## Этап 2

| Пункт ТЗ | Что сделано | Как проверено |
|---|---|---|
| Распознавание голосовых сообщений | Голосовое до 30 секунд и 1 МБ скачивается из Telegram, распознаётся SpeechKit и проходит тот же сценарий, что текст. Пользователь видит распознанную фразу до создания заявки. | Голосовая заявка прошла полный путь и создала сделку №48: Андрей, телефон `+79031234567`. Живой прогон 21.07 (облако клиента): голосовая заявка «клиент с авито… послезавтра в десять утра» создала сделку №70 с источником «Авито» и сроком 23.07.2026 10:00; телефон, продиктованный прописью («восемь девятьсот один…»), распознан и нормализован; голосовое длиннее 30 секунд вежливо отклонено. Тестовые сущности удалены после проверки. |
| Задачи-напоминания через `tasks.task.add` | Просьба напомнить создаёт не сделку, а задачу Bitrix24 с заголовком, дедлайном, ответственным и служебным тегом против дублей. Бот возвращает номер задачи. Задачу создаёт только явная просьба напомнить; приветствия, вопросы и болтовня в CRM не попадают, бот отвечает короткой подсказкой. С раунда 2 напоминание дублируется сообщением бота в Telegram в момент срока, а у сделок со сроком дополнительно создаётся дело CRM с напоминанием (см. этап 1). | Создана задача №4 «Позвонить Ольге» с дедлайном; ответственный — `profiworker24@mail.ru`. Живой прогон 21.07: «напомни позвонить Ольге завтра в 11» → задача №36 «Позвонить Ольге», дедлайн 22.07.2026 11:00 (Владивосток), ответственный id=1, сделка не создана; задача удалена после проверки. На «привет», «что ты умеешь», «спасибо» бот отвечает подсказкой и задачу не создаёт. |
| Категории услуг и все семь статусов | В схеме доступны категории ремонт, перевозки, сборка мебели, электрика, сантехника и прочее. Воронка содержит все семь согласованных стадий; новые сделки получают `NEW`. | Категория «сантехника» записана в сделке №42. Семь стадий сверены в воронке «Услуги на дому» категории `0`. |
| Финансовые поля сделки | Доход, расход и прибыль передаются в Bitrix24; прибыль вычисляется как доход минус расход. | Сделка №42 содержит 5000 / 2000 / 3000 руб. соответственно. |
| Тесты ключевой логики | Офлайн-тестами покрыты разбор, голос, меню, поиск, доступ, дедупликация, черновики, контракты Bitrix24, напоминания (включая планировщик Telegram), правка заявок и обработка сбоев. | Команда `pytest tests/ -q` завершилась результатом `731 passed`; `ruff check app tests` — без замечаний. |
| Финальная инструкция и гайд по расширению | Подготовлены настройка Bitrix24, деплой на Railway, памятка сотруднику и отдельный гайд по добавлению категории, стадии, поля CRM или канала. | Документы находятся в `docs/bitrix-setup.md`, `docs/deploy.md`, `docs/user.md` и `docs/extending.md`; ссылки добавлены в README. |

## Чек-лист передачи

Отправка актуальных коммитов в GitHub клиента и финальный деплой выполняются при
финальной сдаче после подтверждения клиента. До этого они не отмечаются как
завершённые.

- [ ] Отправить актуальную ветку с историей коммитов в репозиторий клиента
  `danilovich25/profiworker24_bot`.
- [ ] После отправки ветки выполнить финальный деплой проекта Railway из
  GitHub-репозитория клиента и проверить запуск бота.
- [x] Telegram-токен принадлежит клиенту и хранится в переменных Railway.
- [x] Входящий вебхук создан в Bitrix24 клиента; его URL не хранится в коде.
- [x] YandexGPT и SpeechKit работают в Yandex Cloud клиента; `YC_API_KEY` и
  `YC_FOLDER_ID` находятся в его Railway-проекте.
- [x] Временный ключ, использовавшийся на стенде разработки, отозван.
- [x] Привязок к чужим серверам, аккаунтам и скрытым сервисам нет.
- [x] Sentry необязателен: при пустом `SENTRY_DSN` бот работает без него.
- [ ] При финальной сдаче передать вместе с исходным кодом инструкции по
  Bitrix24, Railway, работе сотрудника и расширению.
- [x] Гарантийный срок после сдачи — 14 дней.

В объём работ не входят создание самого портала Bitrix24 и обучение сотрудников.
Для самостоятельной настройки портала подготовлена инструкция
`docs/bitrix-setup.md`.
