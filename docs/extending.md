# Как расширять бота

В этом разделе собраны четыре типовых изменения: новая категория услуги,
стадия воронки, поле CRM и новый канал приёма. После правок кода запустите
`ruff check app tests` и `pytest tests/ -q`, затем проверьте сценарий на тестовой
заявке в Bitrix24.

## Новая категория услуг

Список категорий находится в перечислении `Category` в `app/schemas.py`:

```python
class Category(str, Enum):
    repair = "ремонт"
    # ...
```

Чтобы добавить категорию:

1. Добавьте в `Category` новый элемент с уникальным служебным именем и русским
   значением, например `cleaning = "уборка"`.
2. Проверьте разбор фразы с новой услугой и показ карточки заявки.

Остальные места уже используют это перечисление:

- `app/services/llm.py`, функция `_system_prompt`, собирает из `Category`
  допустимые значения для подсказки модели;
- схема `LlmParsedOrder` в `app/schemas.py` ограничивает ответ модели тем же
  перечислением;
- `category_keyboard` в `app/handlers/messages.py` автоматически добавляет
  категорию в кнопки выбора;
- `build_deal_fields` в том же файле добавляет значение в название сделки и
  записывает его в `UF_CRM_SERVICE_CATEGORY`.

Поле категории в Bitrix24 имеет строковый тип, поэтому при добавлении нового
значения менять пользовательское поле CRM не нужно.

## Новый источник заявки

Источники живут в трёх местах и должны совпадать по названию:

1. `app/schemas.py` — значение в `Source` (человекочитаемое название,
   например `telegram_ads = "Реклама в Telegram"`).
2. `app/services/bitrix.py` — пара в `DEAL_SOURCES`: код справочника
   (латиницей, например `TG_ADS`) и то же название. При следующем старте бот
   сам добавит значение в справочник «Источники» портала.
3. `app/services/llm.py` — при необходимости подсказка в `_system_prompt`,
   по каким словам распознавать новый источник в тексте.

Кнопка в опроснике и строка в карточке появятся автоматически: они строятся
из `Source`. Тесты: `tests/test_llm.py` (разбор) и
`tests/test_handlers_messages.py` (кнопки, поле сделки).

## Новая стадия или новое название стадии

Стадии настраиваются в самой CRM. Бот не создаёт и не перестраивает воронку при
запуске.

### Через интерфейс Bitrix24

1. Откройте CRM → Сделки и настройки воронок и стадий.
2. Выберите стандартную воронку «Услуги на дому» — её категория равна `0`.
3. Добавьте стадию, переместите её в нужное место или измените отображаемое
   название существующей стадии.
4. Для конечных стадий правильно задайте результат: успешный для «Завершено» и
   неуспешный для «Отмена».

Первую стадию можно переименовывать, но её код должен остаться `NEW`: функция
`create_deal` в `app/services/bitrix.py` ставит новым сделкам именно
`STAGE_ID="NEW"`. Названия для `/find` и `/last` бот получает через
`crm.status.list`, поэтому изменение только отображаемого названия не требует
правок обработчиков.

### Через REST API

Для автоматизации можно использовать методы `crm.status.list`,
`crm.status.add`, `crm.status.update` и `crm.status.delete`. У стадий стандартной
воронки `ENTITY_ID` равен `DEAL_STAGE`. Не удаляйте `NEW`, пока код создаёт сделки
с этим идентификатором.

Если понадобится отдельная воронка с категорией не `0`, одной настройки в CRM
недостаточно: в `create_deal` потребуется передавать её `CATEGORY_ID` и код
первой стадии этой воронки. Для пользовательских воронок Bitrix24 использует
другие идентификаторы стадий.

## Новое пользовательское поле

Описание полей и их проверка находятся в `app/services/bitrix.py`.

### Поле сделки

1. Объявите константу с полным кодом, например
   `UF_SOURCE = "UF_CRM_SOURCE"`.
2. Добавьте описание в `_UF_FIELDS`. В `FIELD_NAME` указывается имя без
   префикса `UF_CRM_`; выберите подходящий `USER_TYPE_ID` и подписи.
3. Добавьте полный код и ожидаемый тип в словарь для
   `crm.deal.userfield.list` внутри `_REQUIRED_UF_BY_LIST_METHOD`. Так стартовая
   проверка заметит отсутствующее поле или неверный тип.
4. Заполните поле в `build_deal_fields` из `app/handlers/messages.py`.
5. Если значение должно извлекаться из сообщения, добавьте его в `ParsedOrder`
   и `LlmParsedOrder` в `app/schemas.py`, в `_system_prompt` из
   `app/services/llm.py`, в карточку `preview_text` и в пошаговое редактирование.

При следующем старте `ensure_uf_fields` попробует создать поле через
`crm.deal.userfield.add` и проверит результат через
`crm.deal.userfield.list`.

### Поле контакта

Порядок тот же, но описание добавляется в `_CONTACT_UF_FIELDS`, а ожидаемый тип
— в проверку `crm.contact.userfield.list`. Значение нового контакта нужно
добавить в словарь `fields` функции `resolve_contact` в
`app/services/bitrix.py`. Если поле должно обновляться и у найденного контакта,
добавьте его также в вызов `crm.contact.update` этой функции.

`build_deal_fields` формирует только сделку и поля контакта не заполняет.

## Новый канал приёма заявок

Текущие точки входа находятся в `app/handlers/`:

- `messages.py` принимает обычный текст;
- `voice.py` распознаёт голосовое и передаёт полученный текст дальше;
- `search.py` обслуживает поиск и последние заявки;
- `start.py` обслуживает команды и главное меню.

Общая функция запуска заявки — `handle_order_text` в
`app/handlers/messages.py`. Она выполняет защиту от повторов, разбор текста,
уточняющие вопросы и создание карточки. Новый Telegram-источник — например,
подпись к документу — лучше подключать отдельным обработчиком и передавать в
`handle_order_text`:

- объект входящего `Message`;
- текущий `FSMContext` и `Database`;
- нормализованный текст заявки;
- `user_id` отправителя;
- `dedup_key`, полученный от middleware;
- клиент `bitrix`.

Новый роутер экспортируйте из `app/handlers/__init__.py` и поставьте в кортеже
`routers` до `messages_router` и `fallback_router`, если его фильтр более
специфичен, чем обычный текст. Порядок здесь важен: команды и специальные типы
сообщений должны обрабатываться раньше общего текстового маршрута.

Для внешнего источника, который не является Telegram-сообщением, не следует
имитировать `Message`. Вынесите общий прикладной сценарий из обработчика в
сервисный слой, а адаптер нового канала пусть преобразует входные данные в ту
же структуру `ParsedOrder`. При этом сохраните две действующие гарантии:
постоянный ключ источника для защиты от дублей и явное подтверждение перед
созданием сделки.
