# Настройка Bitrix24

Бот работает через входящий вебхук Bitrix24. Перед деплоем нужно создать вебхук,
подготовить стандартную воронку сделок и убедиться, что у портала доступен REST API.

## 1. Проверьте доступ к REST API

Для облачного Bitrix24 в зоне `.ru` работа локальных вебхуков и REST API требует
действующей подписки Битрикс24 Маркетплейс. Условия зависят от редакции и доменной
зоны портала, поэтому перед настройкой проверьте их в разделе «Мой тариф».

Справка Bitrix24:

- [как создать вебхук](https://helpdesk.bitrix24.ru/open/20886106/);
- [когда нужна подписка Маркетплейс](https://helpdesk.bitrix24.ru/open/24577700/).

Если раздел создания вебхуков недоступен или REST-запросы блокируются, сначала
решите вопрос с тарифом или подпиской. Без REST бот сможет отвечать в Telegram,
но не сможет работать с контактами, сделками, полями и задачами.

## 2. Создайте входящий вебхук

Вебхук следует создавать из учётной записи администратора портала. На рабочем
портале этот пользователь также является ответственным за задачи-напоминания
(в коде используется Bitrix ID `1`).

1. В Bitrix24 откройте «Приложения» или «Маркет» → «Разработчикам».
2. Выберите создание вебхука и тип «Входящий вебхук».
3. Укажите понятное название, например `ProfiWorker24 Telegram bot`.
4. Выдайте вебхуку права на следующие разделы:
   - CRM (`crm`) — контакты, сделки, стадии и пользовательские поля;
   - Задачи (`task`) — создание и поиск задач-напоминаний.
5. Сохраните вебхук.
6. Скопируйте базовый URL вида
   `https://ваш-портал.bitrix24.ru/rest/1/секретный_код/`.

В интерфейсе также может показываться URL с методом в конце, например
`crm.deal.list.json`. В `BITRIX_WEBHOOK` нужна только базовая часть, которая
заканчивается секретным кодом и косой чертой.

URL вебхука даёт доступ к данным портала. Не публикуйте его, не добавляйте в
репозиторий и не отправляйте в открытый чат. Если URL стал известен посторонним,
пересоздайте вебхук и замените значение переменной на Railway.

Боту достаточно scope `crm,task`: методы `user.*` он не вызывает. У рабочего
вебхука сейчас указан расширенный scope `crm,disk,task,user`; лишние `disk` и
`user` можно убрать при следующем пересоздании вебхука.

## 3. Настройте воронку «Услуги на дому»

Бот создаёт сделки в стандартной воронке Bitrix24 — категория `0`. Поле
`CATEGORY_ID` он не передаёт, а начальную стадию задаёт кодом `NEW`. Поэтому
нужно переименовать и настроить именно стандартную воронку, а не создавать для
бота отдельную пользовательскую воронку.

В CRM откройте сделки, перейдите к настройке воронок и стадий и задайте название
воронки «Услуги на дому». Стадии должны идти в таком порядке:

| № | Стадия | Что означает |
|---:|---|---|
| 1 | Новая заявка | Заявка только что создана ботом. Код стадии — `NEW`. |
| 2 | Связались с клиентом | Сотрудник связался с заказчиком. |
| 3 | Назначен исполнитель | Ответственный за работу выбран. |
| 4 | В работе | Исполнитель приступил к заявке. |
| 5 | Ожидание оплаты | Работа выполнена или завершается, ожидается оплата. |
| 6 | Завершено | Успешная финальная стадия. |
| 7 | Отмена | Неуспешная финальная стадия. |

Коды остальных шести стадий могут быть назначены порталом: бот от них не
зависит. Он всегда создаёт новую сделку в `NEW`, а дальнейшее перемещение по
воронке выполняют сотрудники или роботы Bitrix24.

## 4. Пользовательские поля

При каждом старте бот вызывает `ensure_uf_fields` из
`app/services/bitrix.py`. Функция пытается создать недостающие поля, а затем
проверяет их имена и типы. Повторный запуск безопасен: уже существующие поля не
создаются заново.

### Поля сделки

| `FIELD_NAME` при создании | Код поля в сделке | Тип | Назначение |
|---|---|---|---|
| `TG_MSG_ID` | `UF_CRM_TG_MSG_ID` | строка | Постоянный ключ сообщения Telegram. По нему бот не создаёт повторную сделку после повторной доставки или сбоя связи. |
| `EXPENSE` | `UF_CRM_EXPENSE` | число (`double`) | Расход по заявке в рублях. |
| `PROFIT` | `UF_CRM_PROFIT` | число (`double`) | Прибыль в рублях: доход минус расход. |
| `SERVICE_CATEGORY` | `UF_CRM_SERVICE_CATEGORY` | строка | Категория услуги, например «сантехника». Используется для фильтров и отчётов. |

Доход отдельным пользовательским полем не создаётся. Он записывается в штатное
поле суммы сделки `OPPORTUNITY`.

### Поля контакта

| `FIELD_NAME` при создании | Код поля в контакте | Тип | Назначение |
|---|---|---|---|
| `TG_DRAFT_ID` | `UF_CRM_TG_DRAFT_ID` | строка | Ключ черновика. Позволяет найти уже созданный контакт после сбоя и не создать второй. |
| `ORG` | `UF_CRM_ORG` | строка | Название организации клиента; по нему работает поиск `/find`. |

Если поле уже заведено вручную, его тип должен совпадать с таблицей. При
отсутствующем поле, неверном типе или недостаточных правах бот пишет в лог
`Поля CRM в Bitrix24 не готовы — отключаю CRM` и не создаёт неполные сделки.

### Справочник источников

Источник заявки бот пишет в штатное поле сделки `SOURCE_ID`. На старте он сам
доводит справочник CRM → «Источники» до нужных значений: добавляет «Авито»
(`AVITO`), «Форпост» (`FORPOST`), «Сарафанное радио» (`SARAFAN`) и
переименовывает штатный `OTHER` в «Прочее». Если справочник обновить не
удалось (нет прав), бот продолжает работать: значение просто не будет подписано
в карточке сделки, о проблеме останется запись в логе.

### Дела-напоминания

У сделки со сроком бот создаёт дело (`crm.activity.todo.add`) с напоминанием в
момент срока. Ответственным ставится пользователь из `BITRIX_RESPONSIBLE_ID`
(по умолчанию `1`) — push мобильного приложения уходит именно ответственному,
поэтому это должен быть пользователь заказчика, а не технический аккаунт.

## 5. Передайте URL приложению

На Railway откройте сервис бота → `Variables`, создайте переменную:

```text
BITRIX_WEBHOOK=https://ваш-портал.bitrix24.ru/rest/1/секретный_код/
```

После сохранения переменной выполните новый деплой. При успешном старте в логах
не должно быть сообщения об отключении CRM. Затем отправьте тестовую заявку,
нажмите «Создать» и проверьте:

- создан или найден контакт;
- создана сделка в воронке «Услуги на дому» на стадии «Новая заявка»;
- номер в ответе бота совпадает с ID сделки;
- категория и финансовые значения попали в поля из таблицы выше.

Полная настройка Railway описана в [инструкции по деплою](deploy.md).
