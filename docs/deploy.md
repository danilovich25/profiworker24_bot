# Развёртывание на Railway

Проект запускается в Docker-контейнере и получает сообщения через Telegram long
polling. Публичный домен Railway ему не нужен. Для постоянной SQLite-базы нужен
отдельный том, смонтированный в `/data`.

## Что подготовить

До начала развёртывания должны быть готовы:

- репозиторий `danilovich25/profiworker24_bot` на GitHub с актуальной веткой;
- аккаунт Railway владельца проекта;
- Telegram-бот и его токен от [@BotFather](https://t.me/BotFather);
- входящий вебхук Bitrix24 с правами `crm` и `task`;
- каталог и сервисный аккаунт Yandex Cloud с доступом к YandexGPT и SpeechKit;
- Telegram ID сотрудников, которым разрешено пользоваться ботом.

Настройка вебхука и воронки описана в
[отдельной инструкции](bitrix-setup.md).

## 1. Создайте проект из GitHub

1. Войдите в Railway под аккаунтом владельца проекта.
2. Нажмите `New Project` → `Deploy from GitHub Repo`.
3. Если Railway ещё не подключён к GitHub, разрешите ему доступ к нужному
   аккаунту и репозиторию.
4. Выберите `danilovich25/profiworker24_bot` и рабочую ветку.
5. Выберите `Add Variables`, чтобы не запускать бота с пустой конфигурацией.

Railway найдёт `Dockerfile` в корне и соберёт образ самостоятельно. Команда
запуска уже задана в образе: `python -m app.main`.

## 2. Добавьте переменные окружения

Откройте сервис → `Variables`. Значения можно добавлять по одному или вставить
через `Raw Editor` в формате `ИМЯ=значение`.

| Переменная | Что указать | Где взять |
|---|---|---|
| `BOT_TOKEN` | Секретный токен Telegram-бота | В диалоге с @BotFather: команда `/token`, затем выбрать бота. |
| `BITRIX_WEBHOOK` | Базовый URL входящего вебхука, оканчивающийся `/` | В карточке входящего вебхука Bitrix24. Подробности — в `bitrix-setup.md`. |
| `YC_API_KEY` | API-ключ сервисного аккаунта | Yandex Cloud → сервисный аккаунт → создание API-ключа. Ключ должен иметь доступ и к YandexGPT, и к SpeechKit. |
| `YC_FOLDER_ID` | Идентификатор каталога, в котором разрешены облачные сервисы | Yandex Cloud → страница каталога; ID отображается в его свойствах. |
| `ALLOWED_TG_IDS` | Telegram ID сотрудников через запятую, точку с запятой или пробел | Каждый сотрудник может посмотреть свой числовой ID через информационного Telegram-бота; перед добавлением сверьте имя и ID. |
| `ALLOW_ALL_USERS` | `false` на проде | Явный аварийный режим без списка доступа. Значение `true` допустимо только на закрытом стенде. |
| `DB_PATH` | `/data/bot.db` | Постоянное значение для Railway; файл будет храниться на подключённом томе. |
| `SENTRY_DSN` | DSN проекта Sentry или пустая строка | Необязательная интеграция. Если мониторинг не используется, оставьте пустым. |
| `TZ` | `Asia/Vladivostok` | Рабочий часовой пояс компании. Он используется для слов «завтра», «через час» и сроков. |
| `BITRIX_RESPONSIBLE_ID` | ID пользователя-владельца портала (обычно `1`) | Bitrix24 → профиль сотрудника, число в адресе страницы. Ответственный за задачи и дела-напоминания: mobile-push уходит именно ему. |

Пример без секретных значений:

```text
BOT_TOKEN=<токен Telegram>
BITRIX_WEBHOOK=<URL входящего вебхука>
YC_API_KEY=<API-ключ Yandex Cloud>
YC_FOLDER_ID=<ID каталога Yandex Cloud>
ALLOWED_TG_IDS=123456789,987654321
ALLOW_ALL_USERS=false
DB_PATH=/data/bot.db
SENTRY_DSN=
TZ=Asia/Vladivostok
BITRIX_RESPONSIBLE_ID=1
```

Не вставляйте реальные токены в файлы репозитория, сообщения об ошибках или
скриншоты. На Railway секреты должны храниться только в `Variables`.

## 3. Подключите постоянный том

1. На схеме проекта нажмите правой кнопкой по сервису и выберите
   `Attach Volume` — либо добавьте Volume через меню создания ресурса.
2. Подключите том к сервису бота.
3. Укажите `Mount Path`: `/data`.
4. Убедитесь, что `DB_PATH` равен `/data/bot.db`.

В SQLite хранятся ключи обработанных сообщений, черновики и защитные записи от
повторного создания сделок и задач. Без тома база исчезнет при новом деплое, и
локальная защита от дублей потеряет историю.

Запускайте один экземпляр сервиса. Проект использует Telegram long polling и
один локальный файл SQLite; несколько реплик для этой конфигурации не нужны.

## 4. Выполните первый деплой

Примените подготовленные изменения и нажмите `Deploy`. После сборки откройте
`Deployments` → активный деплой → `View Logs`.

При успешном запуске видны строки:

```text
Запускаю polling
Run polling for bot ...
```

Первая строка выводится самим приложением, вторая — aiogram. Также в логах могут
появиться сообщения о создании полей Bitrix24. Не должно быть ошибок:

- `BOT_TOKEN не задан`;
- `Поля CRM в Bitrix24 не готовы — отключаю CRM`;
- ошибок авторизации Telegram, Bitrix24 или Yandex Cloud;
- ошибок создания или открытия `/data/bot.db`.

## 5. Проверьте работу

1. Откройте Telegram-бота и отправьте `/start` с разрешённого аккаунта.
2. Отправьте текстовую тестовую заявку и нажмите «Создать».
3. Проверьте, что бот вернул номер заявки, а контакт и сделка появились в
   Bitrix24 на стадии «Новая заявка».
4. Надиктуйте короткую голосовую заявку и убедитесь, что бот показал
   распознанный текст.
5. Выполните `/find <номер заявки>` и `/last`.
6. Отправьте фразу с напоминанием и проверьте задачу в Bitrix24.

Если `/start` не отвечает, сначала проверьте `ALLOWED_TG_IDS`, затем логи
активного деплоя. Публичный URL или настройка Networking для этого бота не
требуются.

## Обязательно на проде

### Закройте доступ списком сотрудников

Заполните `ALLOWED_TG_IDS` реальными числовыми ID. Допустимы запятая, точка с
запятой и пробелы. Пустое значение работает по принципу fail-closed: бот не
пускает никого. Непустое значение без единого числового ID считается ошибкой
конфигурации и останавливает запуск.

`ALLOW_ALL_USERS=true` открывает бота всем. Этот режим предназначен только для
временной отладки на закрытом стенде; на проде оставьте `false`. Постороннему
пользователю бот сообщает об отказе не чаще одного раза в сутки, остальные его
сообщения молча игнорируются.

### Храните базу на томе

Используйте одновременно `DB_PATH=/data/bot.db` и том с точкой монтирования
`/data`. Одной переменной без тома недостаточно: файловая система контейнера
заменяется при деплое.

### Явно задайте часовой пояс

Укажите `TZ=Asia/Vladivostok`. Без корректного часового пояса относительные
сроки могут быть рассчитаны не по местному времени.

## Как заменить токен Telegram

Токен нужно заменить, если он попал посторонним или бот передаётся другому
владельцу.

1. Откройте [@BotFather](https://t.me/BotFather).
2. Чтобы отозвать скомпрометированный токен, отправьте `/revoke` и выберите
   нужного бота. Для получения действующего токена используйте `/token`.
3. Скопируйте новый токен один раз и не сохраняйте его в репозитории.
4. В Railway откройте сервис → `Variables` и замените значение `BOT_TOKEN`.
5. Примените изменения. Если Railway не запустил новый деплой автоматически,
   откройте последний деплой и выберите `Redeploy`.
6. Дождитесь строк `Запускаю polling` и `Run polling for bot ...`, затем
   проверьте `/start`.

После `/revoke` старый токен больше не работает. Возвращать его в переменные
окружения не нужно.
